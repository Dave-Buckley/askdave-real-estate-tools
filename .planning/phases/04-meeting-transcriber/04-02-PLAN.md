  <action>
Create `src/main/transcriber-server.ts` that exports functions: `startServer()`, `stopServer()`, `getServerState()`, and `onStateChange(cb)`.

**HTTP Server (Node.js built-in `http` module):**
- `startServer()` creates an HTTP server on port 0 (OS picks a free port). After listening, determines the machine's local WiFi IP address using `os.networkInterfaces()` (pick the first non-internal IPv4 address). Constructs the full URL (`http://{ip}:{port}`) and stores it.
- GET `/` serves an inline HTML page (the phone recorder -- see below).
- GET `/status` returns JSON `{ connected: true }` for health check.

**WebSocket Server (use Node.js built-in `ws`-free approach via `http` upgrade):**
Actually, use the `ws` package -- it's the simplest Node.js WebSocket library and already works with Electron's Node.js. Install it: `npm install ws` (add to dependencies). Or -- since this is main process Node.js code, we can use `ws`. BUT to avoid adding a dependency, use a simpler approach:

REVISED: Use the raw `http` module with manual WebSocket upgrade. This avoids adding a `ws` dependency. The WebSocket protocol for this use case is simple (phone sends binary audio frames, desktop sends text control messages). Implement a minimal WebSocket handler:
- Parse the Sec-WebSocket-Key header, compute Sec-WebSocket-Accept
- Handle binary frames (opcode 0x02) for audio data
- Handle text frames (opcode 0x01) for JSON control messages from phone
- Send text frames back for control (pause confirmation, stop acknowledgment)

Actually, this is too complex to hand-roll reliably. Use `ws` package instead -- it is tiny, has no native deps, and is the standard for Node.js WebSocket servers.

```bash
npm install ws
npm install -D @types/ws
```

**WebSocket Protocol:**
- Phone connects via `ws://{ip}:{port}`
- Phone sends JSON text messages: `{ type: 'state', state: 'recording' | 'paused' | 'stopped' }` and `{ type: 'elapsed', seconds: number }`
- Phone sends binary messages: raw Float32Array PCM audio chunks (16kHz mono) during recording
- When phone sends `{ type: 'stopped' }`, the phone also sends the final audio as one large binary message with `{ type: 'audio-complete' }` text message after it
- Desktop tracks state and relays to renderer via callback

SIMPLER APPROACH for audio transfer: Rather than streaming audio chunks in real-time (complex, needs reassembly), use a two-phase approach:
1. Phone records locally using MediaRecorder API, accumulates audio in memory
2. Phone sends state updates (recording/paused/stopped/elapsed) as JSON over WebSocket so desktop shows live status
3. When phone stops recording, phone converts the recorded audio to 16kHz mono Float32Array PCM, then sends it as a single binary WebSocket message
4. Desktop receives the binary message, converts it to Float32Array, and provides it to the caller via callback

This is simpler and more reliable than streaming.

**Phone Recorder HTML Page (served inline from `startServer`):**
The page is a self-contained HTML file with inline CSS and JavaScript (no external dependencies, no build step). Requirements:
- Mobile-friendly: viewport meta, large touch-friendly buttons, dark theme matching the desktop app
- Works on iOS Safari AND Android Chrome (MediaRecorder API)
- Big Start button initially, then Pause/Resume and Stop buttons during recording
- Shows elapsed time during recording
- WebSocket connection to the server for state sync and audio transfer
- On stop: converts recorded audio to 16kHz mono Float32Array using OfflineAudioContext for resampling, sends as binary over WebSocket
- Status indicators: "Connected to desktop", "Recording...", "Paused", "Sending audio..."
- No external JS/CSS -- everything inline in the HTML string
- Audio capture: `navigator.mediaDevices.getUserMedia({ audio: true })` with MediaRecorder for recording, then AudioContext for 16kHz conversion on stop

**Audio conversion on phone (in the HTML page JS):**
MediaRecorder produces webm/opus or mp4/aac depending on browser. On stop, decode the blob to AudioBuffer using `AudioContext.decodeAudioData()`, then resample to 16kHz mono using an OfflineAudioContext, extract the Float32Array from the resulting AudioBuffer, and send over WebSocket as binary (transferring the ArrayBuffer).

NOTE on iOS Safari: MediaRecorder may not be available on older iOS versions, but iOS 14.6+ supports it. For maximum compatibility, the page should check for MediaRecorder support and show a clear error if unavailable.

**Server lifecycle:**
- `startServer()` returns `Promise<{ port: number, url: string }>`
- `stopServer()` closes all WebSocket connections, closes the HTTP server, clears all in-memory audio data
- `getServerState()` returns current `TranscriberStatus`
- `onStateChange(cb: (status: TranscriberStatus) => void)` registers a callback for state changes (phone connected, recording state changed, audio received)
- When audio is received from phone, store it as `Float32Array` in module-level variable (in-memory only, no disk). The callback notifies with the audio data.
- Export a `getReceivedAudio(): Float32Array | null` function that returns the audio and then nulls the reference (consume-once pattern, ensures GC)

**Important privacy/security notes:**
- No HTTPS needed -- this is local network only (http://192.168.x.x)
- No authentication needed -- the QR code is the "auth" (user must physically scan it)
- Audio buffer is nulled after consumption (getReceivedAudio returns and clears)
- Server binds to 0.0.0.0 to be accessible from phone on same WiFi

**Error handling:**
- If no WiFi/LAN IP found, throw an error with message "No local network found. Make sure your computer and phone are on the same WiFi network."
- WebSocket disconnect: update state to 'waiting' (phone disconnected), keep server running so phone can reconnect by scanning QR again
- Multiple connections: only allow one phone connection at a time. Reject additional connections.
